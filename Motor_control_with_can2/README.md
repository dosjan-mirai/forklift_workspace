# Motor_control

STM32CubeIDE firmware project for controlling an RS00 motor driver over CAN on an `STM32F746ZG`.

This repository is not just a protocol library drop. It contains:
- a full CubeIDE target project for the STM32F746ZG board
- a reusable RS00 CAN helper library in `Core/Inc/rs00.h` and `Core/Src/rs00.c`
- a project-specific test and bring-up application in `Core/Src/main.c`

## Overview

The current firmware does the following:
- initializes `CAN1`, `USART3`, GPIO, Ethernet, I2C, and USB peripherals generated by CubeMX
- configures CAN for the RS00 motor bus
- performs a simple startup sequence: read `VBUS`, enable active reporting, read key parameters, switch to current mode, enable the motor, and hold `Iq = 0`
- continuously processes CAN feedback and prints status over UART
- exposes two debugger-triggered test commands through Live Expressions

The reusable `rs00` library exists and is suitable for integration into other projects. The checked-in `main.c` still uses local `RobStride_*` helpers for the active test harness, so treat this repository primarily as a working firmware project, not a minimal library example.

## System Model

At a high level, the control chain is:

```text
STM32 host application
  -> CAN transport
  -> RS00 motor driver control mode
  -> internal current / speed / position loops in the drive
  -> motor + load mechanics
```

The STM32 does not implement a full field-oriented controller here. It acts as the supervisory host:
- sends mode changes and setpoints
- reads feedback frames and diagnostic data
- triggers validation routines
- enforces a conservative startup state

This distinction matters when tuning. Some behavior is set by the host update pattern in `main.c`, but much of the closed-loop behavior is inside the RS00 drive itself.

## Hardware Assumptions

- MCU: `STM32F746ZG`
- CAN peripheral: `CAN1`
- CAN pins: `PD0` = `CAN_RX`, `PD1` = `CAN_TX`
- UART debug port: `USART3` at `115200 8N1`
- Default CAN IDs:
  - host ID: `0xFD`
  - motor ID: `0x01`
- CAN timing in the current project:
  - prescaler `3`
  - BS1 `13TQ`
  - BS2 `2TQ`
  - SJW `1TQ`

Bus wiring requirements:
- MCU connected through a proper CAN transceiver
- correct `CANH` / `CANL` polarity
- exactly two `120 ohm` terminations, one at each end of the bus
- common ground between nodes
- matching bitrate on every CAN node

## Startup Behavior

On boot, `main.c` currently performs this sequence:
1. initialize peripherals
2. print `RS00 STM32 port init`
3. read the motor `VBUS` parameter
4. enable active feedback reporting with a `10 ms` period
5. read core parameters
6. force current mode
7. send motor enable
8. write `Iq = 0.0 A`

After initialization, the main loop:
- drains CAN RX frames
- handles debugger-triggered test commands
- sends a safe keep-alive command every `10 ms`
- polls key motor parameters every `200 ms`
- prints compact runtime status over UART when feedback is available

Typical UART status line:

```text
mode=<...> run=<...> fault=0x.. pos=<...> vel=<...> tq=<...> temp=<...> vbus=<...> mech=<...> iq=<...>
```

Field meaning:
- `mode`: feedback status byte from the motor
- `run`: active run mode reported by the drive
- `fault`: fault bitfield reported by the drive
- `pos`: position estimate
- `vel`: velocity estimate
- `tq`: torque estimate
- `temp`: motor temperature
- `vbus`: bus voltage
- `mech`: mechanical position parameter
- `iq`: q-axis current feedback

## Control Theory

The RS00 drive exposes several high-level control modes. They are not equivalent. Each one changes where the main control loop lives and what quantity your host is commanding.

### Operation Mode

This is the MIT-like command path exposed by `RS00_SendOperationControl(...)`.

Conceptually:

```text
tau_cmd = Kp * (q_ref - q) + Kd * (dq_ref - dq) + tau_ff
```

Where:
- `q_ref`: desired position
- `q`: measured position
- `dq_ref`: desired velocity
- `dq`: measured velocity
- `Kp`: stiffness-like gain
- `Kd`: damping-like gain
- `tau_ff`: feedforward torque

Engineering interpretation:
- increasing `Kp` makes the motor resist position error more aggressively
- increasing `Kd` adds damping and helps reduce overshoot
- `tau_ff` biases the actuator with a direct torque request

Use this mode when:
- you want compliant joint behavior
- you need position and velocity shaping together
- you are experimenting with leg, arm, or haptic-style control

Main risk:
- high `Kp` and `Kd` can create oscillation, chatter, or noisy torque behavior, especially with backlash or light loads

Example:

```c
RS00_SetMode(&motor, RS00_MODE_OPERATION);
RS00_Enable(&motor);

// position = 1.0 rad, desired velocity = 0, moderate stiffness/damping
RS00_SendOperationControl(&motor, 0.0f, 1.0f, 0.0f, 20.0f, 1.5f);
```

### Current Mode

Current mode commands `Iq`, which is the torque-producing current component. In practice:

```text
torque approximately proportional to Iq
```

This is the simplest mode for first motion testing because:
- the commanded variable is direct and intuitive
- it is easier to ramp safely from `0 A`
- it exposes whether the bus, enable sequence, and drive state are working

Use this mode when:
- bringing up a new setup
- checking direction and load response
- testing torque-producing capability carefully

Main risk:
- too much commanded current produces sudden torque immediately

Example:

```c
RS00_SetMode(&motor, RS00_MODE_CURRENT);
RS00_Enable(&motor);
RS00_SetCurrentLimit(&motor, 2.0f);
RS00_SetCurrent(&motor, 0.3f);
```

### Velocity Mode

Velocity mode commands angular speed in `rad/s`. The drive typically closes the speed loop internally and uses current control beneath it.

Conceptually:

```text
speed error -> speed controller -> current request -> motor torque
```

Useful when:
- the application is mostly continuous rotation
- you care about speed tracking more than exact position
- you want easier host-side control than a full external position loop

Important parameters in this project:
- speed target: `RS00_PARAM_SPD_REF`
- acceleration limit: `RS00_PARAM_ACC_RAD`
- current limit: `RS00_PARAM_LIMIT_CUR`

Main risk:
- aggressive acceleration with a loose current limit can make startup and reversals harsh

Example:

```c
RS00_SetMode(&motor, RS00_MODE_VELOCITY);
RS00_Enable(&motor);
RS00_SetCurrentLimit(&motor, 3.0f);
RS00_SetVelocityAccel(&motor, 5.0f);
RS00_SetVelocity(&motor, 6.0f);   // rad/s
```

### CSP Position Mode

CSP means cyclic synchronous position. The host updates the target position repeatedly and the drive tracks it as a streamed trajectory.

Use this mode when:
- the STM32 or upstream controller owns the motion trajectory
- you need continuous target updates
- coordinated multi-axis or robotics motion is the goal

Key idea:
- trajectory authority stays on the host side
- the drive handles lower-level tracking

Important parameters:
- position target: `RS00_PARAM_LOC_REF`
- speed limit: `RS00_PARAM_LIMIT_SPD`
- torque limit: `RS00_PARAM_LIMIT_TRQ`

Main risk:
- if host updates are discontinuous, motion can become jerky

Example:

```c
RS00_SetMode(&motor, RS00_MODE_CSP_POSITION);
RS00_Enable(&motor);
RS00_SetSpeedLimitCSP(&motor, 8.0f);
RS00_WriteFloat(&motor, RS00_PARAM_LIMIT_TRQ, 8.0f);
RS00_SetPositionCSP(&motor, 0.8f);   // rad
```

### PP Position Mode

PP means profile position. The host sends a destination and motion limits, and the drive generates the move profile internally.

Use this mode when:
- you want simple point-to-point positioning
- the host should issue high-level commands, not stream every intermediate setpoint
- internal profile generation is acceptable

Important parameters:
- max speed: `RS00_PARAM_VEL_MAX`
- acceleration: `RS00_PARAM_ACC_SET`
- target position: `RS00_PARAM_LOC_REF`

Compared with CSP:
- PP is easier to use
- CSP gives more direct trajectory control

Example:

```c
RS00_SetMode(&motor, RS00_MODE_PP_POSITION);
RS00_Enable(&motor);
RS00_WriteFloat(&motor, RS00_PARAM_LIMIT_TRQ, 8.0f);
RS00_SetPositionPP(&motor, 1.2f, 4.0f, 6.0f);
```

## Mode Selection Guide

Choose a mode based on the controlled quantity:
- choose Current mode when the main question is "how much torque-producing effort should I request?"
- choose Velocity mode when the main question is "how fast should it spin?"
- choose CSP when the host owns the trajectory sample-by-sample
- choose PP when the drive can plan the move internally
- choose Operation mode when you need compliant servo behavior with explicit stiffness and damping terms

Practical recommendation for bring-up:
1. start in Current mode with very low `Iq`
2. move to Velocity mode with conservative acceleration and current limit
3. test CSP or PP at small motion amplitudes
4. tune Operation mode last

## Live Expressions

The current firmware exposes these debugger variables:
- `g_live_cmd`
- `g_live_apply`

To run a command:
1. set `g_live_cmd`
2. set `g_live_apply = 1`

Supported commands in the current `main.c`:
- `15`: run the multi-mode demo sequence
- `16`: run a max-current hold test

### Command 15: All-Mode Test

This sequence steps through:
1. Operation mode
2. Current mode
3. Velocity mode
4. CSP position mode
5. PP position mode
6. return to current mode with `Iq = 0`

UART markers include:
- `[TEST] all_modes begin`
- `[TEST] operation`
- `[TEST] current`
- `[TEST] velocity`
- `[TEST] csp`
- `[TEST] pp`
- `[TEST] all_modes done`

What this test is useful for:
- verifies mode switching works end-to-end
- confirms that the drive accepts each class of command
- provides a fast sanity check after wiring or firmware changes

What it does not prove:
- safe tuning under full load
- thermal limits
- long-duration stability
- application-specific control quality

### Command 16: Max Torque / Max Current Test

This routine:
- prepares current mode
- sets current limit to `12.0 A`
- sets torque limit to `14.0`
- commands `Iq = 12.0 A`
- holds for `30000 ms`
- returns current to zero and sends stop

Use this only when the motor is mechanically safe to load.

## Tuning Workflow

A reasonable engineering sequence for this project is:

1. verify the physical layer first
2. verify feedback and diagnostics
3. test low-current motion
4. open up speed and position modes gradually
5. tune dynamic behavior last

Detailed workflow:

### 1. Physical Layer Check

Before commanding motion:
- check CAN transceiver wiring
- confirm two `120 ohm` terminations only
- verify common ground
- verify power stage and motor power are in expected range

### 2. CAN Health Check

On a healthy bus, expect:
- no bus-off
- `LEC = 0`
- `TEC = 0`
- `REC = 0`

If UART shows repeated `ACK_ERR`:
- motor may not be powered
- node ID may be wrong
- bitrate may not match
- transceiver path may be broken

### 3. Low-Risk Torque Check

Start with Current mode and a small command such as `0.2 A` to `0.5 A`.

Goal:
- confirm direction, enable behavior, and basic responsiveness

Do not jump directly to the max-current test unless the mechanism is restrained or otherwise safe.

### 4. Velocity Mode Tuning

Increase in this order:
1. small target speed
2. moderate current limit
3. moderate acceleration

Observe:
- overshoot
- reversal harshness
- acoustic noise
- current draw

### 5. Position Mode Tuning

For CSP and PP:
- start with small angular targets
- keep speed limits low
- keep torque limits conservative
- only then increase travel and speed

### 6. Operation Mode Tuning

Tune `Kp` and `Kd` incrementally:
- increase `Kp` until tracking becomes acceptably stiff
- add `Kd` until overshoot and ringing reduce
- use `tau_ff` only when feedforward torque is actually justified by the application

If the motor becomes noisy or oscillatory:
- reduce `Kp`
- reduce `Kd`
- reduce target step size
- inspect backlash, compliance, and sensor noise

## Reusable RS00 Library

The RS00 helper library is in:
- `Core/Inc/rs00.h`
- `Core/Src/rs00.c`

Main capabilities:
- CAN startup and filter configuration
- RS00 extended-ID frame construction
- enable, stop, clear-fault, zero, and save-parameter commands
- parameter read/write helpers for `float`, `u8`, `u16`, and `u32`
- high-level helpers for:
  - current mode
  - velocity mode
  - CSP position mode
  - PP position mode
  - MIT-style operation control
- receive parsing into `RS00_State`
- CAN error/status reporting through `RS00_GetCanDiag()`

Minimal integration pattern:

```c
RS00_Handle motor;

RS00_Init(&motor, &hcan1, 0xFD, 0x01);
RS00_StartCan(&motor);
RS00_EnableActiveReport(&motor, 10);
RS00_SetMode(&motor, RS00_MODE_CURRENT);
RS00_Enable(&motor);
RS00_SetCurrent(&motor, 0.0f);

while (1)
{
  RS00_ProcessRx(&motor, 32U);
}
```

Read-back example:

```c
RS00_ReadParam(&motor, RS00_PARAM_VBUS);
HAL_Delay(10);
RS00_ProcessRx(&motor, 8U);

float vbus = motor.state.vbus_v;
uint8_t run_mode = motor.state.run_mode;
float iq = motor.state.iq_fbk_a;
```

Suggested host loop pattern for real applications:

```c
while (1)
{
  RS00_ProcessRx(&motor, 32U);

  // Update commands at a controlled rate.
  // Keep the rate deterministic if using CSP or MIT-like control.

  HAL_Delay(1);
}
```

For higher-performance behavior, replace `HAL_Delay` pacing with a deterministic scheduler or timer-driven control update.

## CAN Diagnostics

The reusable library provides:
- `RS00_GetCanDiag(...)`
- `RS00_LecToString(...)`

Healthy bus indicators:
- `LEC = 0`
- `TEC = 0`
- `REC = 0`
- `BOFF = 0`

Common failure patterns:
- `ACK_ERR`: missing peer ACK, wrong bitrate, bad wiring, or motor absent
- `BIT0_ERR` with `BOFF = 1`: physical layer or bitrate mismatch

Additional practical interpretation:
- good CAN health but no motion usually points to enable/state/mode issues rather than wiring
- valid feedback with unstable motion usually points to control parameter choices or mechanical setup

Diagnostic example:

```c
RS00_CanDiag diag;

RS00_GetCanDiag(&motor, &diag);

if (diag.bus_off || (diag.lec != 0U))
{
  const char *lec = RS00_LecToString(diag.lec);
  // send to UART / debugger log
}
```

## Safety Notes

- Always begin in a mechanically safe condition with the load restrained or de-coupled.
- Treat current commands as torque-producing commands.
- Increase limits gradually, not in one jump.
- Re-check the active run mode before assuming a command type is valid.
- Do not use the max-current routine as a general bring-up step.
- If oscillation starts, reduce command amplitude and gains immediately.

## Troubleshooting

### No CAN Feedback

Check:
- transceiver present and powered
- CAN bitrate match
- CAN IDs correct
- motor node powered and connected

### CAN Looks Healthy But Motor Does Not Move

Check:
- enable command actually sent
- run mode is what you expect
- current, speed, or position limit is not set too low
- fault bits in UART output

### Motion Is Jerky

Likely causes:
- CSP targets updated too coarsely
- acceleration or torque limits too aggressive
- mechanical friction or backlash
- Operation-mode gains too high

### Immediate Fault Or Bus-Off

Check:
- physical wiring
- termination
- supply stability
- ground reference
- CAN timing on all nodes
